# Feature Plan: Risk Profile Questionnaire (FR12.1)

**Feature ID:** FR12.1  
**Status:** ✅ Completed  
**Title:** Risk Profile Questionnaire  
**User Story:** As a user, I want to answer a guided questionnaire about my financial situation and risk tolerance, so that the system can help me determine my appropriate risk profile.

---

## 1. Objective

This document outlines the full-stack design and implementation plan for the "Risk Profile Questionnaire" feature. The goal is to provide a clean, secure, and intuitive questionnaire wizard that persists answers and metadata in the database, with proper column-level encryption in desktop mode to maintain user privacy. The questionnaire utilizes the validated 13-question Grable & Lytton (1999) Financial Risk Tolerance scale.

## 3. Backend Specification

### 3.1. Database Schema
We will create a new table `user_risk_profiles` to store the answers and classification details.

| Column Name     | Data Type     | Constraints                          | Description                                       |
| --------------- | ------------- | ------------------------------------ | ------------------------------------------------- |
| `id`            | `UUID` (GUID) | `PRIMARY KEY`                        | Unique identifier.                                |
| `user_id`       | `UUID` (GUID) | `FOREIGN KEY (users.id)`, `UNIQUE`   | One-to-one link to the user.                      |
| `answers`       | `LargeBinary` | `NOT NULL` (Desktop) / `String`      | **Encrypted JSON** of responses using `EncryptedString`. |
| `score`         | `Integer`     | `NULLABLE`                           | Numerical score calculated from responses.        |
| `risk_category` | `String`      | `NULLABLE`                           | Conservative, Moderate, Growth, or Aggressive.    |
| `updated_at`    | `TIMESTAMP`   | `NOT NULL`, `SERVER_DEFAULT(now())`  | Last submission time.                             |

### 3.2. API Endpoints
All endpoints require authentication via `get_current_user`.
*   `GET /api/v1/risk/`
    *   Retrieves the current user's profile.
    *   Response: `UserRiskProfile` schema (returns 404 if not found).
*   `POST /api/v1/risk/`
    *   Creates or updates the user's answers, calculates the score/category, and returns the profile.
    *   Request Body: `UserRiskProfileCreate` schema.

## 4. Frontend Specification

### 4.1. Questionnaire Wizard
*   A multi-step card layout that tracks state step-by-step.
*   Enforces answer validation per step before allowing the user to proceed.
*   Uses React Query mutations to save the answers to the backend.
*   **INR Currency Localization:** All options use standard Indian Rupee (`₹`) denominations (e.g. `₹10,00,000`, `₹50,000`).
*   **localStorage State Persistence:** Saves the active question step and chosen answers locally. Navigating away or refreshing the page mid-survey will not reset progress.
*   **Onboarding Auto-Routing:** The main `/dashboard` page automatically checks if the user has completed a risk profile; if not, they are auto-redirected to `/risk-profile`.

### 4.2. Results View
*   Displays the user's score (range 13 to 47), profile, and an indicative asset allocation chart.
*   Provides a "Retake" button to redirect back to the wizard (which clears `localStorage`).

## 5. Verification & Test Plan

### 5.1. Automated Tests
*   **Backend Integration Tests:** `backend/app/tests/api/v1/test_risk.py` verifies GET/POST requests, Pydantic validations, database constraints, scoring ranges (Conservative <=18, Moderate <=28, Growth <=32, Aggressive >32), and encryption checks.
*   **Frontend Tests:** Verification of UI renders, navigation button states, and `DashboardPage` query client mocks.

### 5.2. Manual Verification
*   Complete the questionnaire with cautious answers (Score <= 18) and verify the profile maps to "Conservative".
   *   Complete the questionnaire with aggressive answers (Score >= 33) and verify the profile maps to "Aggressive".
*   Retake the questionnaire and verify that the database updates the record without creating duplicate entries.
